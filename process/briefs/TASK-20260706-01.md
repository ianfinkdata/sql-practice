# TASK-20260706-01: Bronze full-row export pack (external BI tooling)

GOAL: Produce a runnable query pack that exports every row of all 14 `oakhaven` bronze
tables to flat files, for Ian to load into Power BI and explore locally alongside a local
Claude session. This is a full-fidelity, grounded bronze data source — not a profiling
sample. Distinct from the existing bronze query pack (`medallion/a_bronze/B01-B06`), which
profiles/reconciles rather than exports.

CONTEXT: IDX-004 (schema snapshot — table list, PKs, exact row counts), IDX-012 (bronze
baseline pack — row-count/enum-census ground truth to verify the export against),
`grounding/medallion-spec.md` §Reproducibility, `grounding/lessons.md` RULE-001/002/010
(deterministic ORDER BY, Windows connection command, PowerShell Get-Content pipe form).

DEFINITIONS REQUIRED: none. This is bronze — raw rows, no business logic applied, no DEF
citations needed (bronze is read-as-is per A1 in IMPLEMENTATION_PLAN.md).

DELIVERABLE: `outputs/TASK-20260706-01/exports/`:
- One `.sql` file per table (14 total), named `E01_<table>.sql` … `E14_<table>.sql`, each a
  single `SELECT * FROM oakhaven.<table> ORDER BY <primary_key>;` — full columns, full rows,
  **no LIMIT** (this is the one sanctioned exception to "add LIMIT to exploratory queries" —
  a full export is the explicit point of this deliverable). Deterministic order via the PK
  per RULE-001. Standard RUN header comment matching the existing bronze pack's format
  (see `medallion/a_bronze/B01_row_counts.sql` for the exact template).
- A single `run_all_exports.ps1` that loops over all 14 files and runs each via the standard
  Windows mysql invocation (`process/mysql-setup.md`), redirecting `--batch` output to
  `exports/csv/<table>.tsv` (tab-separated, header row included — mysql client's native
  `--batch` format, not hand-rolled CSV, so embedded tabs/newlines/NULLs are handled safely
  per the client's own escaping rather than a fragile custom quoting scheme).
- `EXPORT_GUIDE.md` covering: how to run the script, expected row count per table (cross-
  reference `grounding/schema.md`'s exact counts so Ian can sanity-check the output row-by-
  row-count), a note that `--batch` renders SQL NULL as the literal string `NULL` (Power BI's
  Text/CSV connector needs this mapped back to null, not left as text — flag it explicitly
  since it's an easy footgun), and step-by-step Power BI import instructions (Get Data →
  Folder, pointed at `exports/csv/`, delimiter = Tab, first row as headers, NULL-string
  handling per the note above).

DONE MEANS: All 14 tables covered (matches `grounding/schema.md`'s table list exactly, no
extras, none missing); every export query has a deterministic ORDER BY on its PK; RUN headers
follow the established convention exactly (PowerShell Get-Content pipe, not `-e` string
quoting, not `<` redirection — RULE-010); `EXPORT_GUIDE.md`'s row-count table matches
`grounding/schema.md`'s "Exact row counts" section number-for-number; no other project file
touched.

CONSTRAINTS: Read-only against `oakhaven` (SELECT only, no writes anywhere — this task
produces query files and a runner script, it does not execute anything against a live DB in
this session; execution happens locally). This cannot be executed or captured in a session
without DB access (same constraint as TASK-20260705-02) — that's expected and fine, since
there's no "expected output" to capture here beyond the row counts already in
`grounding/schema.md`; the deliverable is the query pack + runner + guide, not a capture.

QA PLAN: Ian runs `run_all_exports.ps1` locally, diffs the resulting row counts per file
against `grounding/schema.md`, and spot-checks a couple of PK columns for correct ordering.
